## Задание

Цель работы - настроить процедуру периодического резервного копирования базы данных, сконфигурированной в ходе выполнения лабораторной работы №1, а также разработать и отладить сценарии восстановления в случае сбоев.

Узел из предыдущей лабораторной работы используется в качестве основного. Новый узел используется в качестве резервного. В сценариях восстановления используется копия данных, полученная на первом этапе данной лабораторной работы.

Требуется выполнить:

### Этап 1. Резервное копирование

- настроить резервное копирование с основного узла на резервный по схеме: первоначальная полная копия + непрерывное архивирование WAL;
- включить режим архивирования WAL;
- настроить копирование WAL на резервный узел с помощью `scp`;
- создать первоначальную резервную копию с помощью `pg_basebackup`;
- перенести базовую копию на резервный узел с помощью `rsync`;
- оценить объем резервных копий спустя месяц при условиях:
  - средний объем новых данных в БД за сутки: `200 MB`;
  - средний объем измененных данных за сутки: `650 MB`.

### Этап 2. Потеря основного узла

Необходимо восстановить работу СУБД на резервном узле, продемонстрировать успешный запуск сервера и доступность данных.

### Этап 3. Повреждение файлов БД

Необходимо:

- симулировать сбой удалением директории любого табличного пространства со всем содержимым;
- проверить работу СУБД и доступность данных;
- перезапустить СУБД и проанализировать результат;
- выполнить полное восстановление данных на основном узле из резервной копии;
- разместить восстановленные данные в другой директории, так как исходный `PGDATA` считается недоступным.

### Этап 4. Логическое повреждение данных

Необходимо:

- добавить в каждую таблицу 2-3 новые строки;
- зафиксировать время;
- удалить каждую вторую строку в одной из таблиц;
- выполнить восстановление данных на основном узле способом `pg_dump` на резервном узле с последующим применением файла на основном узле;
- продемонстрировать результат и выполнить анализ.


2. Настройка параметров сервера PostgreSQL на основном узле:

```bash
  --tablespace-mapping='${HOME}/nym69=${HOME}/backup/tblspc/nym69'
```

```sql

```

6. Перенос полной копии на резервный узел через `rsync`.

Ключевой фрагмент сценария `stage1_backup.sh`:


## Расчет объема резервных копий

В данной работе используется схема:

- одна полная физическая копия;
- последующее непрерывное архивирование WAL.

Предположим, что минимальный объем архивируемых WAL за сутки примерно равен сумме объема новых и измененных данных:

```text
Vwal_day = 200 MB + 650 MB = 850 MB
```

За 30 суток:

```text
Vwal_month = 850 * 30 = 25500 MB
```

Переведем в гигабайты:

```text
25500 / 1024 ≈ 24.90 GB
```

Итоговый объем хранения через месяц:

```text
Vtotal = Vbase + 25500 MB
```

где:

- `Vbase` - размер полной физической копии;
- `25500 MB` - оценка объема архивных WAL за 30 суток.

Если, например, исходная полная копия имеет размер `1.2 GB`, то:

```text
Vtotal ≈ 1.2 GB + 24.9 GB = 26.1 GB
```

### Анализ

Можно сделать следующие выводы:

- основной вклад в объем резервных копий дает не базовая копия, а архив WAL;
- чем выше интенсивность изменений, тем быстрее растет архив;
- рассчитанное значение является нижней оценкой;
- в реальной системе объем WAL будет больше из-за full page image, изменений индексов, `VACUUM`, фоновых процессов и служебных операций.

Следовательно, при практическом планировании объема диска необходимо закладывать дополнительный запас.

## Этап 2. Потеря основного узла

В этом сценарии предполагается полная недоступность основного узла. Для восстановления используется резервный узел и копия данных, полученная на этапе 1.

1. Создает новый каталог данных `FAILOVER_PGDATA=${HOME}/failover_pgdata`.
2. Копирует в него базовую копию из каталога `${HOME}/backup/base`.
3. Добавляет параметры восстановления:


4. Создает файл `recovery.signal`.
5. Запускает PostgreSQL на резервном узле на порту `9099`.

```bash
restore_command = 'cp ${HOME}/archive/%f %p'
recovery_target_timeline = 'latest'
recovery_target_action = 'promote'
```

Проверка результата на резервном узле:

Результат показывает, что кластер успешно поднят на резервном узле, восстановление завершено, а данные доступны для чтения.

## Этап 4. Логическое повреждение данных

Для демонстрации логического восстановления была выбрана таблица `products`, поскольку она наглядна и не содержит внешних ключей, которые усложняли бы частичное восстановление.

На первом шаге в каждую таблицу были добавлены новые строки.

Команда запуска подготовительного SQL-сценария на основном узле:

```bash
cd /Users/skadibtw/ddss/lab2
psql -v ON_ERROR_STOP=1 -p 9099 -d bigbluecity -f scripts/stage4_prepare.sql
```

Сценарий выполняет:

- вывод количества строк до изменений;
- добавление по 2 строки в таблицы `customers`, `products`, `stores`, `sales`;
- вывод содержимого таблицы `products` после вставки;
- фиксацию времени восстановления `recovery_target_time`;
- удаление каждой второй строки из таблицы `products`;
- вывод итогового состояния таблицы `products` после ошибки.

Фрагмент `stage4_prepare.sql`:

```sql
INSERT INTO products (name, category, price) VALUES
('Lab2 SSD', 'Электроника', 9900.00),
('Lab2 Mouse', 'Аксессуары', 1900.00);

SELECT clock_timestamp() AS recovery_target_time \gset
\echo :recovery_target_time

DELETE FROM products
WHERE id IN (
  SELECT id
  FROM (
    SELECT id, row_number() OVER (ORDER BY id) AS rn
    FROM products
  ) s
  WHERE rn % 2 = 0
);
```

Пример ожидаемого фрагмента вывода:

```text
=== RECOVERY TARGET TIME ===
2026-03-21 18:42:11.512345+03

=== AFTER DELETE ===
 id |      name       |   category    |  price
----+-----------------+---------------+---------
  1 | Ноутбук         | Электроника   | ...
  3 | Наушники        | Аксессуары    | ...
  5 | Lab2 SSD        | Электроника   | 9900.00
```

Видно, что часть строк была удалена ошибочной операцией `DELETE`.

### Восстановление данных

Для восстановления используется резервный узел. На нем разворачивается временный кластер из базовой копии, затем выполняется PITR до времени непосредственно перед ошибочным удалением. После этого из полученного состояния выгружается таблица `products` с помощью `pg_dump`, а затем дамп вручную переносится на основной узел и применяется через `pg_restore`.

Сценарий запускается локально на резервном узле:

```bash
cd /Users/skadibtw/ddss/lab2
TARGET_TIME='2026-03-21 18:42:11.512345+03' bash scripts/stage4_restore_from_reserve.sh
scp ${HOME}/transfer/products_before_delete.dump postgres0@pg125:${HOME}/transfer/products_before_delete.dump
```

Сценарий делает следующее:

1. На резервном узле разворачивает временный кластер `RESERVE_STAGE4_PGDATA=${HOME}/stage4_pgdata`.
2. Указывает параметры восстановления:

```text
restore_command = 'cp ${HOME}/archive/%f %p'
recovery_target_time = '2026-03-21 18:42:11.512345+03'
recovery_target_inclusive = 'true'
recovery_target_action = 'promote'
```

3. Запускает PITR на резервном узле.
4. Выполняет:

```bash
pg_dump -h localhost -p 9191 -d bigbluecity -Fc -t public.products -f ${HOME}/transfer/products_before_delete.dump
```

5. После этого дамп вручную переносится на основной узел.
6. На основном узле затем выполняется:

```bash
mkdir -p ${HOME}/transfer
pg_restore --clean --if-exists --no-owner --no-privileges \
  -h localhost -p 9099 -d bigbluecity -t public.products \
  ${HOME}/transfer/products_before_delete.dump
```

Проверка результата на основном узле:

```bash
psql -v ON_ERROR_STOP=1 -h localhost -p 9099 -d bigbluecity -c 'TABLE products;'
```

Ожидаемый результат: таблица `products` возвращается в состояние до ошибочного удаления, включая только что добавленные строки.

### Анализ

Этот способ восстановления удобен в ситуациях, когда:

- повреждена не вся БД, а отдельный объект;
- нужно минимизировать простой рабочей системы;
- необходимо восстановить данные на конкретный момент времени.

Комбинация физической базовой копии и архивных WAL позволила получить на резервном узле точное состояние БД перед ошибкой, а логический `pg_dump` дал возможность перенести только нужную таблицу обратно на основной узел.

## Ответы на вопросы для защиты

### 1. Резервные копии: назначение; локальные и удаленные; полные и инкрементные; холодные и горячие

Резервные копии нужны для восстановления после отказов оборудования, ошибок администратора, логического удаления данных и полной потери узла.

- локальные копии хранятся на том же сервере и быстро восстанавливаются, но не защищают от потери самого узла;
- удаленные копии хранятся на другом сервере и защищают от полной потери основного узла;
- полные копии содержат все данные целиком;
- инкрементные копии содержат только изменения относительно предыдущего состояния;
- холодные копии создаются при остановленной СУБД;
- горячие копии создаются без остановки сервера.

### 2. Сравнение подходов SQL Dump, FS Level Backup, Continuous Archiving

`SQL Dump`:

- логический уровень;
- удобен для отдельных БД и таблиц;
- хорошо переносится между версиями PostgreSQL;
- медленнее на больших объемах;
- восстановление тоже занимает много времени.

`FS Level Backup`:

- физическое копирование файлов кластера;
- быстро восстанавливается;
- подходит для больших БД;
- требует согласованного состояния файловой системы.

`Continuous Archiving`:

- основан на архивировании WAL;
- позволяет выполнять PITR;
- обеспечивает минимальную потерю данных;
- требует аккуратного хранения WAL и контроля за их объемом.

На практике наиболее надежной является комбинация полной физической копии и непрерывного архивирования WAL.

### 3. Как рассчитать объем пространства для хранения резервных копий, от чего зависит

Объем хранения зависит от:

- размера полной копии;
- интенсивности изменений;
- частоты создания копий;
- срока хранения;
- числа поколений резервных копий;
- коэффициента сжатия;
- накладных расходов WAL и служебных данных.

Базовая формула:

```text
V = Nfull * Vfull + Ndays * Vwal_day + Vreserve
```

### 4. Как рассчитать требуемую производительность канала для выполнения резервных копий по сети, от чего зависит

Необходимая пропускная способность канала оценивается формулой:

```text
Bandwidth >= V / T
```

где:

- `V` - объем передаваемых данных;
- `T` - допустимое окно резервного копирования.

На требуемую производительность влияют:

- размер базовой копии;
- объем WAL;
- допустимое время передачи;
- шифрование и сжатие;
- сетевые задержки;
- скорость дисковой подсистемы на обоих узлах.

### 5. Какие факторы оказывают влияние на скорость восстановления, почему

На скорость восстановления влияют:

- размер базовой копии;
- объем WAL, который необходимо применить;
- скорость чтения резервной копии;
- скорость записи на целевой диск;
- число табличных пространств;
- количество индексов и общая сложность структуры БД;
- необходимость восстановления на конкретный момент времени;
- производительность CPU и объем доступной памяти.

Чем больше WAL нужно проиграть после восстановления базовой копии, тем дольше длится процесс запуска кластера после аварии.

## Вывод

В ходе выполнения лабораторной работы была настроена схема резервного копирования PostgreSQL по модели «полная физическая копия + непрерывное архивирование WAL» с переносом данных на резервный узел. Были подготовлены локальные сценарии для аварийного запуска базы данных на резервном сервере, полного восстановления основного узла после частичного физического повреждения (потери одного табличного пространства) и восстановления логически поврежденных данных через PITR на резервном узле с последующим ручным переносом дампа и `pg_restore` на основном узле.

В результате была подтверждена работоспособность выбранной схемы резервного копирования и показано, что комбинация `pg_basebackup` и архивных WAL позволяет восстанавливать как весь кластер, так и отдельные объекты базы данных.
